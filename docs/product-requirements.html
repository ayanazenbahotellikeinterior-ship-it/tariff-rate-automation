1. ビジョン

「輸入案件ごとに発生する『粗利表の関税率更新』を、複数ファイル横断の手作業から解放する」 ツールを提供する。 輸入許可証PDF・PL・PO の3種の資料を横断して関税率を割り出し、現状 1週間 かかる作業を 30分以下 に短縮する。

本プロダクトは ローカルファースト・設定駆動・監査可能 を3つの柱とし、 将来的なクラウド統合や GUI 化よりも、まずは 確実に動く CLI ツール で日常業務を変えることを優先する。

2. ターゲットユーザー

属性内容
主たるユーザー輸入案件を担当する個人（社内オペレーション担当）
利用シーン輸入案件（荷受）のたびに発生する粗利表の関税率更新業務

3. 主要機能

  1. 輸入許可証PDF の自動抽出: 1案件あたり最大十数欄ある品目ごとに HSコード・関税率を取得
  2. PL（Excel）の自動抽出: DESCRIPTION と HSコードを取得
  3. PO の自動抽出: 品番・Item・gender・Quality(素材)・Set upマーカー を取得（案件ごとに様式が違うため列マッピング設定で吸収）
  4. 粗利表（スプシ）からの品番取得: Google Sheets API（閲覧）で粗利表の対象品番リストと行順を取得
  5. 三ホップ突合: 品番 → Item/gender/素材 → DESCRIPTION → HSコード → 関税率
  6. 「Set up」展開: TOPS+PANTSのセット品を自動展開し、HS一致を確認
  7. HSコード・関税率の正規化: 桁数差・区切り差・S/Gプレフィックス・FREE表記を吸収
  8. 中間Excel 出力: 3シート構成（結果 / 証跡 / サマリ）で監査可能
  9. 未マッチ理由のレポート: 失敗時は「どのホップで失敗したか」を明示し、人の補完を支援

※ 粗利表AJ列への自動書き込み、PDFのOCR、あいまいマッチ、GUI化は Phase 2 候補（第11節）。

5. 要件

5.1 ビジネス要件

  • 個人利用のため、追加のクラウド利用料・SaaS契約は発生しないこと。
  • Google Sheets API 以外の外部サービス依存を持たないこと（API キーの管理対象を増やさない）。
  • 輸入案件の機密情報（取引先・金額）を外部送信しないこと。すべてローカル処理。

5.2 機能要件（FR・抜粋）

詳細は作業単位の .steering/20260513-tariff-rate-automation/requirements.html 第8節を参照。

FR-CPO/PL/許可証/粗利表 の4ソースを横断して品番ごとの関税率を導出できる

FR-EHSコード・関税率の正規化により異なる表記の差を吸収できる

6. ユーザーストーリー

IDストーリー
US-01輸入担当者として、案件のファイル一式を固定フォルダに置いてスクリプトを1回実行するだけで、関税率の集約結果が得られたい。
US-02輸入担当者として、中間Excelを見て粗利表AJ列の数式中の関税率を書き換えるだけで作業を完了させたい。
US-03輸入担当者として、自動でマッチできなかった品番について「なぜ失敗したか」を知りたい。手動補完の判断材料が欲しいから。
US-04輸入担当者として、Set up（セット品）も自動展開してほしい。手で2行を見比べる手間を減らしたいから。
US-05輸入担当者として、案件ごとにPOの様式が変わっても設定ファイルの調整だけで使い回したい。
US-06輸入担当者として、許可証PDFが20欄を超える規模でも問題なく処理してほしい。
US-07輸入担当者として、誤った関税率が出たときに突合経路を遡って原因を特定したい。

7. 受け入れ条件

本プロダクトを「Phase 1 リリース完了」と見なすには、以下すべてを満たすこと。

  1. サンプル案件（許可証PDF 1件 + PL 1件 + PO 1件 + 粗利表）で実行し、中間Excel が3シート構成で生成される。
  2. マッチ成功品番の関税率値が許可証記載と一致する（実サンプル全件で確認）。
  3. マッチ失敗品番に「どのホップで失敗したか」が記載される。
  4. Set up展開・HS正規化・関税率正規化が期待通り動作する。
  5. 所要時間が 30分以下。
  6. 同一入力で2回実行しても中間Excel の内容が完全一致する。

8. ユースケース図

9. スコープ外（Phase 1 でやらないこと）

  • 粗利表（スプシ）のAJ列への自動書き込み（Phase 2）
  • 輸入許可証PDFのOCR（スキャン画像は対象外）
  • 関税率以外の項目（仕入単価・数量・原産国等）の更新
  • あいまいマッチ・自動学習
  • 定期実行・GUI化
  • 複数案件の同時バッチ処理

10. 制約・前提

  • 実行環境は Python（ローカル Windows PC）に限定。
  • 輸入許可証PDFは税関発行の決まった様式・テキスト選択可能 を前提（OCRは Phase 2）。
  • PL の DESCRIPTION → HSコード、許可証の HSコード → 関税率は原則 1:1 を前提。
  • 粗利表は C列=品番（一意）、AJ列==PRODUCT(AA*, X%) 形式の関税数式、データ3行目以降 を前提。
  • Google Sheets API（閲覧権限のみ）が利用可能な状態である。
  • 本プロダクトの作業単位の詳細は .steering/ 配下のステアリングドキュメントを参照。

11. ロードマップ（Phase 1 / Phase 2）

11.1 Phase 1（本リリース）

  • PDF/Excel 横断抽出 → 三ホップ突合 → 中間Excel 出力
  • 粗利表からの読み取り（Sheets API 閲覧権限）
  • 人が中間Excelを見て、粗利表AJ列の数式を手で書き換える運用

11.2 Phase 2（将来検討）

  • 粗利表AJ列への 自動書き込み（Sheets API 編集権限、数式生成）
  • PL素材サイレントケースの自動判定（HS番号体系からの素材逆引き）
  • あいまいマッチ（語彙辞書で吸収しきれない揺れに対応）
  • 過去案件のHS割当を蓄積した自動学習
  • 定期実行・GUI化
