1. 言語・ランタイム

  • Python 3.11 以上（型ヒント・dataclass・match 文を活用）
  • OS: Windows 10/11（業務PC環境）。WSL は使わずネイティブ実行を前提とする。
  • 仮想環境は venv を使用（.venv/ をプロジェクト直下に配置）。

2. プロジェクト構成

詳細は .steering/.../design.html 第13節 を参照。要点は以下：

tariff-rate-automation/
├─ src/          # 実装コード（モジュールごとに分割）
├─ tests/        # pytest テスト + fixtures
├─ inputs/       # 案件ファイル投入先（.gitignore）
├─ outputs/      # 中間Excel出力（.gitignore）
├─ logs/         # 実行ログ（.gitignore）
├─ secrets/      # service_account.json（.gitignore 必須）
├─ config.yaml
├─ vocabulary.yaml
├─ requirements.txt
└─ README.md
  • 1モジュール = 1責務。reader / normalize / matcher / writer を分離。
  • 純粋関数は normalize_*.py / vocabulary.py / matcher.py に集約。

3. コーディング規約（Python）

3.1 スタイル

  • PEP 8 準拠。フォーマッタは black（line-length 100）。
  • インポート整理は isort（プロファイル black）。
  • 静的解析は ruff（既定ルール + E F W I B UP）。

3.2 型ヒント

  • 公開関数・メソッドは 必ず型ヒントを書く。
  • 戻り値が複数値なら tuple[X, Y] または dataclass。生の辞書を返さない。
  • 外部I/O境界（YAML読込・PDF抽出など）では dataclass または TypedDict で型を明示。

3.3 コメント

  • 「何をしているか」ではなく「なぜそうしているか」を書く。コードが自明な処理にはコメント不要。
  • 業務知識（HSコードの桁構造、許可証PDFの欄区切り規則 等）は docstring に簡潔に書く。

3.4 関数設計

  • 純粋関数を優先（同じ入力 → 同じ出力、副作用なし）。
  • 1関数の行数目安は 50行以内。超える場合は分割を検討。
  • I/O とロジックを混ぜない（reader は読むだけ・matcher は突合だけ）。

5. エラーハンドリング

  • 業務的に「想定内の失敗」（未マッチ等）は 例外で止めず、結果データとして記録。
  • システム的に「続行不可」（設定ファイル不正・認証失敗・必須ファイル欠如）は 明確なメッセージで即終了（非ゼロ exit）。
  • 例外を握りつぶさない。except Exception: pass は禁止。
  • ライブラリ固有の例外は、業務的な意味のある例外に raise ... from e でラップする。

6. ログ出力

  • 標準ライブラリ logging を使用。print() はコンソール出力（サマリ表示）に限定。
  • ログレベル:
    • DEBUG: 内部処理の詳細（行ごとの突合結果等）
    • INFO: 処理の節目（ファイル読込開始/完了、件数）
    • WARNING: 想定外だが続行可能（複数候補ヒット、レイアウト揺れ）
    • ERROR: 続行不可（認証失敗、ファイル欠如）
  • ログファイルは ./logs/run_YYYYMMDD_HHmm.log に出力。
  • 機密情報（取引先名・金額）はログに出さない。

7. 設定ファイル

  • 設定は YAML（config.yaml / vocabulary.yaml）。コードに値をハードコーディングしない。
  • パスはすべて 相対パス（プロジェクトルート基準）。
  • 設定読込時に必須項目をチェックし、欠如時は分かりやすいメッセージで終了。
  • 環境依存値（APIキー・サービスアカウントパス等）は設定ファイル中で 相対パス参照のみ、実物は ./secrets/ に置く（コミットしない）。

8. テスト

8.1 テストフレームワーク

  • pytest を使用。tests/ 配下に test_*.py として配置。
  • カバレッジ計測は任意（必要に応じて pytest-cov）。

8.2 テストの分類

分類対象原則
ユニットテスト純粋関数（normalize_* / vocabulary / matcher 内ロジック）外部I/Oを叩かない。モック・フィクスチャを使用。
reader テスト各 reader モジュールtests/fixtures/ の実サンプル（最小版）で実行。
結合テストmain 全体サンプル一式で実行し、出力Excelの3シート内容を期待値と比較。

8.3 Sheets API のテスト

  • 本物の Sheets API は叩かない。sheet_reader をモックして DataFrame を直接返す。
  • 認証エラー・権限エラーの挙動はモック側で再現する。

8.4 フィクスチャ管理

  • tests/fixtures/ には 機密情報を含まない最小サンプル を配置。
  • 取引先名・金額が含まれる実データはコミットせず、./inputs/ に置いて .gitignore 対象とする。

9. 依存ライブラリ管理

  • 依存は requirements.txt に **バージョンを固定** して記載（pkg==X.Y.Z）。
  • 追加時は理由をコミットメッセージ or PR 説明に書く。
  • 不要になったライブラリは速やかに削除（依存膨張を防ぐ）。
  • 主要ライブラリは design.html 第12節 参照。

10. Git 規約

10.1 ブランチ

  • 個人運用のため、Git Flow までは必須としない。
  • 大きな機能追加・リファクタリングは作業ブランチ feature/<short-name> を切る。

10.2 コミットメッセージ

  • 1行目（subject）は日本語可・50文字以内。
  • 変更の意図が分かるよう「何を・なぜ」を書く。
  • 大きい変更は本文（body）で詳細を補足。空行を1つ入れて段落分け。
例:
permit_pdf_reader: 関税率の FREE 表記を 0% へ正規化

許可証PDF の一部の欄が "G FREE" のように記載されており、
従来の正規表現では数値が取れずに空欄になっていた。
normalize_tariff で FREE → 0% に変換する処理を追加。

10.3 コミット粒度

  • 「動く状態」を保ったまま小さく切る。
  • 機能追加とリファクタリングは別コミット。
  • テスト追加は対象実装と同じコミット または 直後のコミット。

10.4 .gitignore

  • 必須: .venv/, secrets/, inputs/, outputs/, logs/, __pycache__/, *.pyc, .pytest_cache/, ~$*（Excel一時ファイル）

11. セキュリティ・シークレット管理

  • service_account.json 等の認証情報は 絶対にコミットしない。./secrets/ 配下に配置し .gitignore で除外。
  • 万一コミットしてしまった場合: 即座に該当キーを 無効化・再発行。git履歴からの除去は git filter-repo 等で行うが、無効化が最優先。
  • 機密情報（取引先名・金額・個人情報）はログ・コミットメッセージ・コメントに残さない。
  • サンプルデータをドキュメントに含める場合は **マスク** すること（取引先名→「ABC社」、金額→ダミー値等）。

12. ドキュメント運用

  • 恒久的なドキュメント（プロダクト全体）は docs/、作業単位は .steering/[YYYYMMDD]-[題名]/ に置く（CLAUDE.md 参照）。
  • HTML は docs/assets/design-system.css を相対参照。新規HTMLは docs/assets/template.html を雛形にする。
  • 用語は glossary.html に集約し、コード上の命名と整合させる。
  • 図表は SVGインライン または HTML+CSS。外部CDN依存・Mermaid 記法は使わない（オフライン閲覧性のため）。
  • 変更があったら関連ドキュメントを同時に更新（コード変更だけで完了させない）。
