1. 技術スタック

1.1 ランタイム

  • 言語: Python 3.11+
  • OS: Windows 11（開発）/ Linux（コンテナ・本番）

1.3 フロントエンド

  • HTML5 + CSS3 + Vanilla JavaScript（ES2020+）
  • SPA フレームワーク（React/Vue 等）は 採用しない。理由: 画面数6・状態管理が単純・ビルド不要 を優先。
  • CDN・外部ライブラリへの依存は 持たない（オフライン動作可能性のため）。

1.4 インフラ

  • コンテナ: Docker（Dockerfile + docker-compose.yml）
  • HTTPS: ホスティング先のマネージドHTTPS（Cloud Run / Render 等）or Caddy/Nginx + Let's Encrypt
  • 永続化: ファイルシステム（初期）→ SQLite/PostgreSQL（規模拡大時）
  • キャッシュ・キュー: なし（FastAPI BackgroundTasks）→ 規模拡大時に Redis + Celery

2. システム構成（高位）

詳細は .steering/.../design.html 2節 を参照。要点：

  • 3層構成: ブラウザ（HTML/CSS/JS）→ FastAPI（Web層）→ 既存CLIモジュール（業務ロジック層）
  • 業務ロジック層は変更しない（CLI と Web で共通）
  • 状態管理: 初期はファイルシステム + メモリ（ジョブはJSON永続化、セッションはCookie）

4.1 推奨

  • 初期（PoC〜5名利用）: Google Cloud Run（無料枠が大きい・Sheets API と相性良・コールドスタートも許容範囲）
  • 機密性最優先: 社内サーバ（クラウドにファイルを置きたくない場合）
  • 開発フェーズ: ローカル Docker（docker compose up）

4.2 選定時のチェックリスト

  • 機密ファイル（取引先・金額入り）をクラウドに置いて良いか
  • 月額コストの上限
  • HTTPS設定・ドメイン管理の手間
  • 初回デプロイの所要時間
  • 本番URL を社外公開するか（VPN内のみか）
  • 運用負荷（24h稼働・ログ確認頻度）

5. 技術的制約

  • Python のみ: 他言語の混在は避ける（保守者の学習コスト最小化）。
  • 外部 SaaS は最小限: Google OAuth と Google Sheets API のみ。他の SaaS（DB as a Service 等）は使わない。
  • ビルドステップを増やさない: フロントは Webpack/Vite 等を使わない。バンドラ不要のシンプル構成。
  • 機密情報はリポジトリに含めない: .env / secrets/ / 実 PDF/Excel は全て .gitignore。
  • CDN 依存を避ける: ドキュメント・UI ともに自己完結する HTML/CSS/JS のみ。
  • 1人運用前提: 複雑な権限管理・運用自動化は持たない（必要になったら段階的に追加）。

7. セキュリティ・コンプライアンス

7.1 認証

  • Google OAuth 2.0（Authorization Code Flow）
  • 許可ドメインリスト で社外を拒否
  • セッションは HttpOnly + Secure + SameSite=Lax の暗号化Cookie
  • セッションTTL: 8時間

7.2 通信

  • 本番は HTTPS 必須（クラウドはマネージド・社内は Caddy + Let's Encrypt）
  • OAuth コールバックURLも HTTPS

7.3 機密ファイル管理

  • アップロードファイル・結果Excelは 30日で自動削除
  • ユーザー間の隔離: ジョブIDで分離、API はリクエストユーザーが所有者か検証
  • at-rest 暗号化: ホスティング先の機能に依存（Cloud Storage は標準で暗号化）

7.4 シークレット管理

  • 環境変数経由のみ（リポジトリに含めない）
  • 本番: Cloud Secret Manager / Render Env / Docker secrets
  • 開発: .env（.gitignore 必須）

7.5 入力検証

  • ファイル形式（拡張子 + MIMEタイプ）
  • ファイルサイズ（10MB上限）
  • HTML エスケープ（Jinja2 自動）
  • SQL インジェクション: 該当なし（DB未使用）

7.6 ログ・監査

  • 機密情報（取引先名・金額）はログに出さない
  • 認証イベント（ログイン/失敗）・ジョブ操作はサーバログに記録

8. スケーラビリティ方針

8.1 初期リリース（〜5名・週数回）

  • シングルプロセス FastAPI + BackgroundTasks（最大5並列）
  • ファイルシステム永続化（JSON）
  • セッションはメモリ + 暗号化Cookie

8.2 中規模化（〜30名・日数十件）

  • DB 導入（SQLite or PostgreSQL）でジョブ・履歴を永続化
  • セッション保存先を Redis に
  • uvicorn ワーカー複数化

8.3 大規模化（数百名規模）

  • ジョブキューを Celery + Redis に切替
  • ロードバランサ + 複数インスタンス
  • ストレージを Cloud Storage / S3 に移行

8.4 設計上の備え

  • job_store / file_store はインターフェース抽象化されており、実装を差し替え可能
  • job_runner も非同期インターフェース統一で、後から Celery 等への置き換えが容易

9. 監視・ログ

9.1 ログ

  • 標準ライブラリ logging を使用
  • レベル: DEBUG/INFO/WARNING/ERROR（development-guidelines.html 6節）
  • ファイル出力: /data/logs/run_YYYYMMDD_HHmm.log
  • 本番ではホスティング先のログ集約（Cloud Logging / Render Logs）も併用

9.2 ヘルスチェック

  • GET /healthz: 200 OK を返すだけ（コンテナの liveness/readiness 用）

9.3 メトリクス（任意）

  • 初期は不要
  • 規模拡大時に Prometheus エクスポート + Grafana を検討

9.4 アラート

  • 初期は不要（個人運用）
  • チーム規模が大きくなったら、ホスティング先のアラート機能を使う

10. CI/CD（参考）

本リリースは個人運用のため CI/CD は必須ではない。将来導入時の参考として記載。

10.1 ローカル開発フロー

  1. git pull
  2. pip install -r requirements.txt
  3. pytest でユニットテスト
  4. black . && ruff check . でフォーマット・静的解析
  5. docker compose up でローカル動作確認
  6. git commit && git push

10.2 デプロイフロー（候補）

  • Cloud Run: gcloud run deploy（手動）or Cloud Build によるGit連携
  • Render: Git push で自動デプロイ（Connect GitHub）
  • 社内サーバ: git pull && docker compose pull && docker compose up -d を手動 or rsync スクリプト

10.3 GitHub Actions（任意）
