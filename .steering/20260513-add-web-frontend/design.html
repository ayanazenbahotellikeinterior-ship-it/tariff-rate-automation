1. 設計方針

  • 既存 CLI モジュールは変更しない。Web 層は CLI の薄いラッパーとして実装。
  • 非同期ジョブ: アップロード後すぐにレスポンス、処理は別タスクで実行。フロントはポーリングで状態取得。
  • 状態は最小限: 初期リリースはインメモリ + ファイルシステム保存（DB なし）。スケール要件が出たら DB 導入を検討。
  • ホスティング非依存: 設定はすべて環境変数 + YAML、Docker で完結。
  • セキュリティ第一: HTTPS必須、ファイルはユーザー間隔離、シークレットは環境変数のみ。
  • フロントは「シンプル」: 初期はテンプレートエンジン（Jinja2）+ vanilla JS。SPA フレームワークは導入しない。

3. 認証フロー（Google OAuth）

3.1 採用方式

  • OAuth 2.0 Authorization Code Flow（サーバ側でコードを受け取り、サーバ側でトークン取得）
  • ライブラリ: Authlib（FastAPI 統合がシンプル）
  • スコープ: openid email profile（Sheets API は別途サービスアカウント認証で読む）

3.2 フロー

  1. ユーザーが /login にアクセス → Google 認証画面へリダイレクト
  2. 認可後、Google が /auth/callback?code=... へリダイレクト
  3. サーバが code をトークンに交換し、ユーザー情報（email）を取得
  4. ドメイン許可リスト と email を照合。不一致なら 403 を返す
  5. JWT トークン or 暗号化セッション Cookie を発行（HttpOnly, Secure, SameSite=Lax）
  6. 以降のリクエストは Cookie で認証

3.3 ドメイン制限

# config.yaml
auth:
  allowed_domains:
    - "hotellikeinterior.com"
  allowed_emails:               # 個別許可（オプション）
    - "external-partner@example.com"
  session_ttl_seconds: 28800    # 8時間

3.4 サインアウト

  • /logout: Cookie 削除 + Google の revoke エンドポイントを呼ぶ。

4. ジョブモデルと状態遷移

4.1 ジョブの状態

4.2 ジョブのライフサイクル

  1. POST /api/upload でジョブ生成 → 状態: pending
  2. バックグラウンドタスクが受け取り → running
  3. 処理完了 → completed（結果ファイル保存）
  4. 例外発生 → failed（エラー詳細を記録）
  5. 30日経過 → expired（結果ファイル削除）

4.3 非同期実行の実装

  • FastAPI の BackgroundTasks または asyncio.create_task で実行（初期リリース）。
  • 同時実行数の制御: asyncio.Semaphore(5) で最大5並列に制限。
  • 将来規模が大きくなれば Celery + Redis に置き換え可能な抽象化を保つ。

5.5 認証エラー / 認可エラー

  • 未ログインで保護APIにアクセス: 401 Unauthorized

  • ドメイン外でログイン試行: 403 + 説明ページ

6. データモデル

6.1 Job

@dataclass
class Job:
    job_id: str                  # j_YYYYMMDDHHmmssNNNN
    user_email: str
    status: JobStatus            # Enum: pending/running/completed/failed/expired
    spreadsheet_id: str | None
    file_paths: dict             # {"permit": "/uploads/.../permit.pdf", ...}
    result_path: str | None      # /outputs/.../tariff_*.xlsx
    summary: dict | None         # rows数/成功数 etc.
    rows: list[dict] | None      # 結果行（GET /result で返す）
    progress: dict | None        # {step, label, percent}
    error: str | None
    created_at: datetime
    updated_at: datetime
    expires_at: datetime         # created_at + 30日

6.2 永続化

  • 初期リリース: ファイルシステムに JSON で保存（/jobs/{job_id}.json）。
  • 同時アクセスは少ない想定（最大5並列）のためファイルロックで十分。
  • 将来規模が大きくなれば SQLite / PostgreSQL に置き換え。

7. ファイルストレージ設計

7.1 ディレクトリ構成（コンテナ内）

/data/
├─ uploads/{job_id}/
│   ├─ permit.pdf
│   ├─ po.pdf or po.xlsx
│   └─ pl.xls or pl.xlsx
├─ outputs/{job_id}/
│   └─ tariff_YYYYMMDD_HHmm.xlsx
├─ jobs/
│   └─ {job_id}.json
└─ logs/
    └─ run_YYYYMMDD_HHmm.log

7.2 アクセス制御

  • ファイルへの直接 URL アクセスは不可。/api/jobs/{job_id}/download 経由のみ。
  • API はリクエストユーザーが job.user_email と一致するかチェック。

7.3 自動削除

  • 起動時とジョブ作成時に「30日以上前」のジョブをスキャンして削除。
  • cron 的な定期実行は持たない（リクエスト駆動の lazy cleanup）。

7.4 暗号化

  • ファイルシステム自体の at-rest 暗号化はホスティング先（クラウドストレージ / OS）に任せる。
  • 機密性が高いと判明したら Phase 2 でアプリ層 AES 暗号化を検討。

8. フロントエンド設計

8.1 採用方針

  • サーバサイドレンダリング（Jinja2）+ vanilla JS による軽量構成。React/Vue 等の SPA フレームワークは導入しない。
  • 理由: 画面数が少ない（6画面）、状態管理が単純（ジョブのステータスポーリングのみ）、SEO 不要、ビルドステップを増やしたくない。
  • モックアップ（mockup.html）のデザインをそのままテンプレート化。

8.2 主要テンプレート

テンプレート対応画面用途
base.html共通ヘッダ・フッタ・CSS読込
login.htmlSCR-001-login「Googleでログイン」ボタン
upload.htmlSCR-002-upload3ファイルドロップゾーン + 実行ボタン
processing.htmlSCR-003-processingスピナー + 進行状況。JSがポーリング
result.htmlSCR-004-result結果テーブル + サマリ + ダウンロード
history.htmlSCR-005-history過去ジョブ一覧
error.htmlSCR-006-error認証拒否・処理失敗の表示

8.3 静的アセット

  • static/css/app.css: モックアップの CSS をそのまま移植。
  • static/js/upload.js: ドラッグ&ドロップ・ファイル検証・アップロード。
  • static/js/processing.js: 1秒ごとに /api/jobs/{id}/status をポーリング。完了で /result へ遷移。

8.4 アクセシビリティ

  • ボタン・リンクは <button> / <a> を使用（div でない）。
  • フォーム要素には <label> + aria-describedby でヒント付与。
  • 処理中の状態変更は aria-live="polite" でスクリーンリーダーへ通知。
  • キーボード操作: Tab で全要素到達可能、Enter で実行可能。

9. 既存 CLI モジュールの統合

9.1 統合方針

  • CLI モジュールは 変更しない。Web 層から関数呼び出しする。
  • 具体的には tariff_rate_automation パッケージとしてインストール可能にし、from tariff_rate_automation import run_pipeline のように呼ぶ。

9.2 呼び出しインターフェース

# CLI 側に追加する公開API（既存ロジックの薄いラッパー）
def run_pipeline(
    permit_path: Path,
    po_path: Path,
    pl_path: Path,
    spreadsheet_id: str,
    config: Config,
    progress_callback: Callable[[str, int], None] | None = None,
) -> PipelineResult:
    """3ファイル + Sheets から関税率を算出し、Excel と結果サマリを返す。"""
    ...

@dataclass
class PipelineResult:
    summary: dict
    rows: list[dict]
    output_excel_path: Path

9.3 進捗通知

  • Web 層が progress_callback を渡し、各ステップ完了時に呼ばれる。
  • コールバックは Job の progress フィールドを更新 → フロントがポーリングで取得。

9.4 設定の渡し方

  • Web 側は config.yaml を読み、必要箇所だけ上書きして CLI に渡す。
  • 例: spreadsheet_id はリクエストごとに変えられるよう、API パラメータで上書き。

12. デプロイ設計（Docker）

12.1 Dockerfile（要点）

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
ENV DATA_DIR=/data
VOLUME ["/data"]
EXPOSE 8000
CMD ["uvicorn", "web.main:app", "--host", "0.0.0.0", "--port", "8000"]

12.2 docker-compose（ローカル開発用）

services:
  web:
    build: .
    ports: ["8000:8000"]
    environment:
      - GOOGLE_OAUTH_CLIENT_ID=${GOOGLE_OAUTH_CLIENT_ID}
      - GOOGLE_OAUTH_CLIENT_SECRET=${GOOGLE_OAUTH_CLIENT_SECRET}
      - SESSION_SECRET_KEY=${SESSION_SECRET_KEY}
      - ALLOWED_DOMAINS=hotellikeinterior.com
    volumes:
      - ./data:/data
      - ./secrets:/app/secrets:ro

12.3 ホスティング先の候補

詳細は docs/architecture.html で比較。候補は以下：

  • 社内サーバ / 社内PC: Docker さえ動けばOK。VPN必要。コストゼロ。
  • Google Cloud Run: Docker をそのままデプロイ。HTTPS自動。月額数千円〜。
  • Render / Railway: PaaS型。Git push で自動デプロイ。月額無料枠あり。
  • AWS App Runner: Cloud Run と類似。

12.4 HTTPS

  • クラウド (Cloud Run / Render 等) は HTTPS 自動。
  • 社内サーバの場合は Caddy / Nginx + Let's Encrypt のリバースプロキシを推奨。

15. テスト方針

  • 既存 CLI テスト: そのまま維持（pytest）。
  • API テスト: fastapi.testclient で各エンドポイントをテスト。
    • 認証必須エンドポイントへの未ログインアクセスが 401 を返すか
    • 他人のジョブへのアクセスが 403 を返すか
    • ファイル形式不正で 400 を返すか
  • ジョブランナーテスト: パイプラインをモックして状態遷移を確認。
  • OAuthテスト: 本物の Google には繋がず、Authlib のモックでフロー検証。
  • E2Eテスト（手動）: 実サンプル3ファイルをアップロード → 結果に 7.4% が表示されるか目視確認。
