1. 背景と目的

1.1 背景

  • 既存の 関税率自動化 CLI（Pythonローカル実行）は個人の手元では動くが、他のチームメンバーにも使ってほしい。
  • 非エンジニアのメンバーがターミナルでコマンドを打つのは現実的でない。
  • 各メンバーが個別に Python 環境を整備するのは保守負担が大きい。

1.2 目的

既存 CLI を バックエンド（Python モジュール）として再利用 し、ブラウザから操作できる Web フロントエンドを追加する。 チームメンバーが ブラウザでファイルをアップロード → 結果を画面で確認 → 中間Excelをダウンロード できる状態を目指す。

2. 対象ユーザー

属性内容
主たるユーザー輸入案件を担当する自分 + 同チームのメンバー（複数人）
利用シーン輸入案件のたび。各自が担当案件の関税率を確認するために実行
デバイスPC ブラウザ（Chrome / Edge / Safari 最新版）。スマホは対象外
技術リテラシブラウザを日常的に使える。ターミナル・Pythonは使わない前提
非ターゲット社外関係者・取引先

3. ゴール範囲（スコープ）

3.1 含める（IN）

  • FastAPI ベースの Web サーバ（バックエンドAPI）
  • Google OAuth によるユーザー認証（ドメイン制限可能）
  • 3ファイル（許可証PDF / PL / PO）の ブラウザからのアップロード
  • 処理ジョブの非同期実行（アップロード後すぐにレスポンス、結果は後追い）
  • 処理結果（中間Excel）の ブラウザ上プレビュー と ダウンロード
  • 処理履歴の表示（ユーザーごと）
  • 既存 CLI モジュール（reader / matcher / writer 等）の そのまま再利用
  • ホスティング非依存のアーキテクチャ（ローカル / 社内サーバ / クラウドのいずれにも置けるよう設計）

3.2 含めない（OUT）

  • スマートフォン向け UI（PC ブラウザのみ最適化）
  • 粗利表AJ列への自動書き込み（Phase 2 — 別ステアリングで起こす）
  • 細粒度のユーザー権限管理（管理者・一般等の役割分離）。ログインユーザーは全機能利用可
  • 処理結果のクラウド永続化・長期アーカイブ（既定: 30日で自動削除）
  • 多言語化（日本語のみ）
  • Web ベースでの設定ファイル編集 GUI（config.yaml / vocabulary.yaml はサーバ管理者が手で更新）

4. システム構成（高位）

5. 入力（Input）

CLI 版と同じ3ファイル + 粗利表（Sheets）+ 認証情報。

種類取得方法備考
輸入許可証 PDFブラウザからアップロード1ファイル/案件・最大10MB
PL Excelブラウザからアップロード1ファイル/案件・最大10MB
PO PDF/Excelブラウザからアップロード1ファイル/案件・最大10MB
粗利表（スプレッドシート）サーバが Sheets API（閲覧権限）で取得シートIDは設定 or UIで指定
ログイン情報Google OAuth リダイレクトフロー許可ドメイン制限あり

6. 出力（Output）

  • ブラウザ上のプレビュー: 中間Excel「結果」シートを HTML テーブルとして表示（成功品番・関税率・未マッチ理由が一目で分かる）
  • 中間Excelダウンロード: 3シート構成のxlsxファイル（CLI版と同じ仕様）
  • 処理サマリ: 画面上に件数・マッチ率・実行時間を表示
  • 処理履歴: ユーザーごとの過去ジョブ一覧（実行日時・対象品番数・成功率・ダウンロードリンク）

7. ユーザーストーリー

IDストーリー
US-01チームメンバーとして、Googleアカウントでログインして自分の案件を処理したい。アカウント発行の手間を省きたい。
US-02チームメンバーとして、3ファイルをドラッグ＆ドロップで指定して「実行」ボタンを押すだけで処理したい。コマンド入力したくないから。
US-03チームメンバーとして、処理結果を画面上で確認し、必要なら Excel をダウンロードしたい。毎回ダウンロードしなくても大筋を把握したいから。
US-04チームメンバーとして、過去自分が処理した案件の履歴を見返したい。同じ案件を二重処理しないため。
US-05チームメンバーとして、未マッチ理由が画面上で見えるようにしてほしい。Excelを開かなくても判断したいから。
US-06チームメンバーとして、処理に時間がかかる場合は進行状況（ジョブ状態）を見たい。固まっているのか動いているのか分かるように。
US-07運用者として、社外ドメインのGoogleアカウントではログインできないようにしたい。情報漏洩を防ぐため。

8. 機能要件（FR）

ID要件
FR-W01Google OAuth でユーザー認証ができる
FR-W02許可ドメインリスト（YAML設定）に含まれないGoogleアカウントは拒否できる
FR-W03未ログイン状態でAPIエンドポイント・画面にアクセスするとログイン画面にリダイレクトされる
FR-W043ファイル（許可証PDF / PL / PO）をブラウザからアップロードできる（ドラッグ&ドロップ + クリック選択）
FR-W05アップロードファイルの形式（拡張子・サイズ）を検証し、不正なら明示的なエラーメッセージを返す
FR-W06「実行」ボタン押下で処理ジョブを非同期に開始し、ジョブIDを即座に返す
FR-W07ジョブの状態（pending / running / completed / failed）を確認できる
FR-W08処理が完了したら、画面上で結果（成功品番リスト・未マッチ品番リスト・サマリ）を表示できる
FR-W09中間Excel（3シート構成）をブラウザからダウンロードできる
FR-W10ユーザーごとの処理履歴を一覧表示できる（過去30日分）
FR-W1130日経過した結果ファイルは自動的にサーバから削除される
FR-W12処理中・処理完了・処理失敗の状態がブラウザ上で視覚的に区別できる（ステータスバッジ等）
FR-W13処理ログ（既存CLIモジュールが吐くログ）はサーバ側に保存され、ユーザーは閲覧不可（管理者のみ）

9. 非機能要件（NFR）

ID分類要件
NFR-W01パフォーマンスアップロード〜画面上の結果表示まで 5分以内（1案件、許可証16欄規模）
NFR-W02同時実行同時に最大 5ジョブ まで並行処理可能
NFR-W03可用性営業時間内（平日9-18時）の稼働率 99% 以上を目標
NFR-W04セキュリティHTTPS 必須。アップロードファイルはユーザー間で漏洩しない（ジョブIDで隔離）
NFR-W05機密保持アップロードファイル・結果ファイルは サーバ上で暗号化（at-rest） または短期間で削除
NFR-W06互換性Chrome / Edge / Safari の最新2バージョンで動作
NFR-W07レスポンシブPCブラウザ幅 1280px / 1920px で表示崩れなし。スマホは対象外
NFR-W08アクセシビリティWCAG 2.1 AA 準拠（キーボード操作・コントラスト比 4.5:1 以上）
NFR-W09移植性Docker コンテナで動作。ローカル/社内サーバ/クラウドのいずれにもデプロイ可能
NFR-W10保守性既存 CLI モジュールは 変更なし で再利用（Web 層は CLI の薄いラッパー）

10. 受け入れ条件

  1. 許可されたGoogleアカウントでログインでき、許可外ドメインは拒否される。
  2. 3ファイルをドラッグ&ドロップでアップロード→「実行」→ 結果が画面に表示される。
  3. 結果画面で成功品番の関税率・未マッチ品番の理由が一覧で確認できる。
  4. 中間Excel（3シート）がダウンロードでき、CLI 版と同等の内容である。
  5. 過去ジョブの履歴が表示され、再ダウンロード可能（30日以内）。
  6. 1案件の処理が5分以内に完了する（許可証16欄規模）。
  7. HTTPS で通信し、ログインなしの直接アクセスはリダイレクトされる。
  8. Docker で起動でき、ローカル・社内サーバ・クラウドいずれにもデプロイできる構成。

11. スコープ外（やらないこと）

  • スマートフォン向け UI（PCブラウザのみ）
  • 粗利表AJ列への自動書き込み（別 Phase）
  • 管理者ダッシュボード・権限管理（ログインユーザーは全員同権限）
  • 結果の長期アーカイブ（30日で自動削除）
  • 多言語化
  • YAML設定のWeb編集 GUI
  • Webhook・通知（Slack/Email等）

12. 制約・前提

  • バックエンドは 既存 CLI 設計 を そのまま再利用（変更しない）。
  • Web フレームワーク: FastAPI。
  • 認証: Google OAuth 2.0（許可ドメイン制限）。
  • ホスティング先は未確定。Docker コンテナで完結 させて移植性を確保。
  • Python 3.11 以上。
  • 初期リリースは少人数（5名以下）想定。

13. UI影響セクション

UI変更あり（新規UI作成）

本ステアリングは Web フロントエンドの新規追加であり、UI 全体が新規。詳細は docs/ui-specification.html で定義する（同時並行で作成）。

13.1 想定画面（暫定）

画面ID画面名目的
SCR-001-loginログイン画面Google OAuth でログイン
SCR-002-uploadアップロード画面3ファイルを指定して「実行」
SCR-003-processing処理中画面ジョブステータスを表示
SCR-004-result結果画面成功品番・未マッチ品番・サマリの表示、Excelダウンロード
SCR-005-history履歴画面過去ジョブ一覧・再ダウンロード
SCR-006-errorエラー画面処理失敗・認証拒否時の表示

ワイヤフレーム・状態定義・遷移図は docs/ui-specification.html に記載予定。

13.2 ui-specification.html での記載必須項目

  • 画面一覧（上記の確定版）
  • ユーザーフロー / 画面遷移図
  • 各画面のワイヤフレーム（PC、SVG）
  • UI要素一覧（要素名・型・役割・バリデーション・必須/任意）
  • 状態定義（空・読込中・エラー・成功・大画面/中画面）
  • 操作とイベント（ボタン・リンクごとに表形式）
  • アクセシビリティ（WCAG 2.1 AA）
  • 共通コンポーネント（ボタン・フォーム・モーダル・トースト）
  • デザイントークン（色・タイポ・スペーシング）

14. リスク・未確定事項

#項目影響対応方針
RW-01 ホスティング先未確定 初期コスト・運用負荷の見積もり不能 Docker完結で設計し、デプロイ先は architecture.html で複数候補を比較
RW-02 機密ファイルのサーバ保持 情報漏洩リスク 30日自動削除・at-rest暗号化・ユーザー間隔離（ジョブID）
RW-03 Google OAuth クライアントID/シークレットの管理 シークレット漏洩時のリスク 環境変数で注入・secrets/ に同梱しない
RW-04 Sheets API の編集権限（Phase 2 で必要）への移行 権限拡大時の同意フローが変わる 本 Phase は閲覧権限のみ。Phase 2 で再同意を取得する設計に
RW-05 長時間ジョブ（許可証20欄超）のタイムアウト 処理失敗の頻発 非同期ジョブキュー設計でタイムアウト緩和。進行状況表示で UX 確保
RW-06 ファイルアップロード時の進捗バー 大容量ファイル時の UX FastAPI のチャンク受信 + フロント側で進捗表示
