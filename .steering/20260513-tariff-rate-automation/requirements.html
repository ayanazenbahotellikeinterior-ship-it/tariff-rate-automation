1. 背景と目的

1.1 背景（現状の業務）

  • 輸入案件（荷受）のたびに、粗利表（Googleスプレッドシート）の商品ごとの関税率を 手入力で更新 している。
  • 具体的には、AJ列の数式 =PRODUCT(AA*, 9%) の「9%」部分を、当該品番の正しい関税率に **書き換え** ている。
  • 正しい関税率を割り出すには、輸入許可証（PDF）と PL（Excel）を突合し、PO（PDF/Excel）を介して品番と紐づける作業が必要。
  • 1案件あたり 2〜3日、最大1週間 を要する。

1.2 目的

「複数の入力ファイルを横断し、商品ごとの関税率を中間Excelに集約する」工程を自動化し、 手作業を 1週間 → 30分以下 に短縮する。 スプシへの貼付（数式書き換え）は人が中間Excelを見ながら手動で行う（Phase 2 で自動化検討）。

3. ゴール範囲（スコープ）

3.1 含める（IN）

  • 輸入許可証 PDF からの HSコード（税表番号・品目番号）・関税率の自動抽出（1PDFに最大十数欄）
  • PL（Excel）からの DESCRIPTION / HSコードの自動抽出
  • PO（PDF/Excel）からの 品番 / Item / gender / 素材（Quality）の自動抽出
  • 三ホップ突合（品番 → PL DESCRIPTION → 許可証品目番号 → 関税率）
  • 「Set up」（セット品）マーカー検出と PL の TOPS+PANTS への展開
  • HSコード桁数差の正規化（区切り除去・頭6桁での照合）
  • 関税率の正規化（プレフィックス除去・FREE → 0%）
  • 中間Excel（結果 / 証跡 / サマリ の3シート構成）の生成

3.2 含めない（OUT）

  • 粗利表（スプシ）への自動書き込み／自動数式更新（Phase 2 で検討）
  • 輸入許可証PDFのOCR処理（テキスト選択可能PDFのみ対象）
  • 関税率以外の項目（仕入単価・数量・原産国等）の更新

4. 業務フロー（三ホップ突合）

関税率を品番に紐づける情報の流れを以下に示す。

4.1 処理ステップ

  1. 固定入力フォルダ（./inputs/）から全入力ファイルを読み込む。
  2. Google Sheets API（閲覧権限）で粗利表のC列（品番）と行順を取得する。
  3. 各品番について PO から Item / gender / Quality(素材) を取得する。
  4. Item語彙 + gender + 素材 を **部分一致 AND** で PL の DESCRIPTION に対して検索し、対応行を特定。
  5. 「Set up」マーカーがある場合は、TOPS と PANTS の2行を引き、HSコードの一致を確認。
  6. PLのHSコードを 頭6桁に正規化 し、許可証PDFの品目番号（同じく頭6桁正規化）と照合して関税率を取得。
  7. 関税率を正規化（S/G プレフィックス除去、FREE → 0%）。
  8. 粗利表の行順を保ったまま中間Excelの「結果」シートに記載。証跡シートに突合経路、サマリシートに集計を出力。

5. 入力（Input）

5.1 入力ファイル種別

種類形式取得項目備考
輸入許可証 PDF（テキスト選択可・税関の決まった様式） 欄番号 / 品名 / 税表番号 / 品目番号 / 関税率 1PDFに 最大十数欄（実サンプルは16欄／8ページ）
PL（Packing List） Excel DESCRIPTION / HSコード HSコードは 10桁ベタ書き または XXXX.XX-XXXX 形式（混在）
PO（Purchase Order） PDF または Excel（案件ごとに違う） 品番 / Item / gender / Quality(素材) / 「Set up」マーカー 列構成が 案件ごとに違う（要 列マッピング設定）
粗利表 Google スプレッドシート C列（品番）と行順 データは3行目以降。Sheets API（閲覧権限）で取得

5.2 入力ファイル配置

  • 固定フォルダ（例: ./inputs/）に 全入力ファイルを放り込んで一括処理。
  • ファイル種別はファイル名・拡張子・内容ヘッダから自動判定。

5.3 PO の列構成多様性

実サンプル検証で、PO は案件ごとにヘッダ構造が異なることが判明。設定ファイルで以下を吸収する：

  • 品番列名（"No" 等、案件ごとに異なる）
  • Item列名（"Item"）
  • gender列名（"gender" / "Size" 等）
  • 素材を含む列名候補（"Material" / "Quality"）
  • Set upマーカー列名（空欄列 / 専用列）

5.4 許可証PDFの記載パターン

  • 各欄は < NN 欄>統合先欄 のマーカーで区切られる。
  • 関税率は S 4.5% / G 5.3% / G FREE 等のパターン（プレフィックス + 数値% または FREE）。
  • HSコードは 税表番号（4+2桁） と 品目番号（4+2+3桁） の2種類。

6. 出力（Output）

6.1 出力ファイル

  • 中間Excel 1ファイル: ./outputs/tariff_YYYYMMDD_HHmm.xlsx
  • 3シート構成（下記）

6.2 シート1「結果」 — 粗利表貼付用

列内容備考
A: No粗利表の行番号（3始まり）—
B: 品番粗利表C列の品番をそのまま行順保持
C: 関税率5.4% 形式の文字列／FREEは 0% として記載失敗時は空欄
D: 未マッチ理由マッチ失敗時のみ理由を記載成功時は空欄

6.3 シート2「証跡」 — 突合経路（監査用）

1品番＝1行で、PO → PL → 許可証 の各ホップで何にマッチしたかを記録する。Set up展開行はセル内改行で表示。

主な列: 品番 / PO Item / gender / 素材 / マッチ語彙(導出) / PL DESCRIPTION / PL HSコード / HS正規化(頭6桁) / 許可証欄 / 許可証品名 / 許可証品目番号 / 関税率 / ステータス

6.4 シート3「サマリ」 — 処理実行レポート

実行日時 / 読み込んだファイル一覧 / 対象品番数 / マッチ成功率 / 未マッチ理由の内訳件数 / 警告事項を記載。

6.5 未マッチ理由の文言（粒度）

ケース関税率列未マッチ理由列
マッチ成功関税率の値空欄
POに品番が見つからない空欄「POに品番なし」
PL の DESCRIPTION 検索で部分一致行が見つからない空欄「PLにItem/Quality該当なし（検索キー: WOMEN'S+COTTON+...）」
Set upでPLのTOPS+PANTSの HS が不一致空欄「PLでSet up展開時にHS不一致（TOPS=XX, PANTS=YY）」
許可証に該当HSコードがない空欄「許可証に該当HSコードなし（PL=XXXXXXXXXX, 頭6桁=XXXXXX）」

7. ユーザーストーリー

IDストーリー

US-02輸入担当者として、中間Excelの「結果」シートの関税率列を見て、粗利表のAJ列の数式 =PRODUCT(AA*, 9%) の 9% を該当の率に書き換えるだけで作業を完了させたい。

US-07輸入担当者として、突合の経路（証跡）を後から追跡可能にしてほしい。誤った関税率が出たときに原因を遡れるように。

10. 受け入れ条件

  1. サンプル案件（許可証PDF 1件 + PL 1件 + PO 1件 + 粗利表）で実行し、中間Excel が3シート構成で生成される。
  2. 「結果」シートの関税率列を見て、粗利表のAJ列の数式の 9% を該当の率に書き換えるだけで更新が完了する。
  3. マッチ成功した品番について、関税率の値が許可証PDFの記載と一致している。
  4. マッチ失敗した品番について、未マッチ理由列に「どのホップで失敗したか」が記載されている。
  5. Set up（セット品）の場合、TOPS+PANTS の HS が一致していれば自動で1つの関税率を返し、不一致なら未マッチ理由に明記される。
  6. HSコードの桁数差・区切り文字差（PL 10桁 vs 許可証 4+2桁／4+2+3桁）が吸収されて正しく照合される。
  7. 関税率の S/G プレフィックス・FREE 表記が正しく正規化される。
  8. 1案件の所要時間が 30分以下 である（実測）。
  9. 同じ入力で2回実行しても、生成される中間Excelの内容が完全一致する。

11. スコープ外（やらないこと）

  • 粗利表（スプシ）への自動書き込み／自動数式更新（Phase 2）
  • 輸入許可証PDFのOCR（スキャン画像PDFは対象外）
  • 関税率以外の項目（仕入単価・数量・原産国等）の更新
  • PO・PL に対する高度なデータクレンジング・あいまいマッチ（部分一致 AND + 語彙辞書まで。それ以上は次イテレーション）
  • 定期実行（cron 等によるスケジュール起動）
  • GUI（CLI 実行のみ）

12. 制約・前提

  • 実行環境は Python（ローカルPC） とする。
  • 輸入許可証PDFは 税関発行の決まった様式・テキスト選択可能 である。
  • 許可証の HSコード → 関税率は 1:1（実サンプル16欄で確認済み）。
  • PL の DESCRIPTION → HSコードは 原則 1:1（同一DESCRIPTIONで異なるHSが出る稀ケースは未マッチ理由に降格させる方針）。
  • 粗利表は C列で品番を一意に識別・データ3行目以降・AJ列に =PRODUCT(AA*, X%) 形式の関税数式。
  • Google Sheets API（閲覧権限のみ）が利用可能な状態である。
  • PO の列構成は案件ごとに異なる可能性があるため、列マッピングは設定ファイル化する。

14. リスク（実サンプル検証結果）

14.1 検証で解消されたリスク

#項目検証結果
R-01PO と PL の表記揺れ表記揺れではなく 表現粒度の違い と判明。語彙辞書 + 部分一致AND で対応可能。
R-02許可証PDFの抽出方式テーブル抽出 / 正規表現の両方で対応可能。< NN 欄> マーカーで欄を区切り、各キーを引く方針で確定。
R-03粗利表のシート構造C列=品番、AJ列=関税数式、データ3行目以降と判明。設定ファイルで吸収。
R-04Google Sheets API 認証サービスアカウント（閲覧権限）で十分。Phase 2 で編集権限に拡張。

14.2 新たに見つかったリスク

#項目影響対応方針
R-05 PO の列構成が案件ごとに違う 固定スキーマの抽出ロジックが通用しない YAML設定で 列マッピング を案件ごとに切替可能にする
R-06 Set up（セット品）で PL の TOPS と PANTS で HS が一致しないケース 1つの関税率に収束できない 未マッチ理由に明記して人に判断を仰ぐ
R-07 HS正規化の粒度（頭6桁 vs 頭9桁） マッチ率と精度のトレードオフ 既定 頭6桁、設定で切替可能。実測で再評価
R-08 PL の DESCRIPTION に素材が明記されないサイレントケース 部分一致での絞り込み不足 複数候補ヒット時は未マッチ扱いとして人に提示
R-09 関税率の S/G プレフィックスの意味確認 抽出時の解釈に影響 現状は記号を除去して数値%のみ採用。意味は業務確認後に証跡に注記

15. Phase 2（将来検討）

本リリースのスコープ外。実装着手はしないが、設計上の余地は残す。

  • 粗利表 AJ列への自動書き込み: 中間Excelで確定した関税率を、=PRODUCT(AA*, X%) 形式の数式として Sheets API で直接書き込む。認証スコープを「閲覧」から「編集」へ昇格。
  • あいまいマッチ: Item語彙辞書だけで吸収しきれない表記揺れに対し、レーベンシュタイン距離等のあいまいマッチを導入。
  • HS分類の自動学習: 過去案件のHS割当を蓄積し、新規品番に対するHS候補を提示。
  • 定期実行・GUI化: 案件フォルダ監視で自動起動。Web UI で結果を閲覧。
